The idea, in plain language

The mine already measures everything. It just isn't understood fast enough.

Two kilometres underground, three things quietly cause harm: bad air, failing equipment, and mismatched ventilation. None announce themselves clearly. Today a human is expected to watch dozens of live numbers across many zones and connect the dots, every time, all shift. That's exactly the job a machine should help with.

☁

Bad air

Diesel exhaust (CO, NO₂), blasting gases, dust, and falling oxygen. Often you can't smell the danger until it's serious.

⚙

Failing equipment

Pumps, fans, and drills vibrate differently for days before they fail. Miss it, and a $500K breakdown, or worse, follows.

🌀

Wasted ventilation

Fans can be 40%+ of a mine's energy bill. Too much wastes money; too little is a safety risk. Matching airflow to people is hard by hand.

The gap we close

The sensors exist. The brain and the friendly face don't.

Maestro Digital Mine has air-quality stations running in Vale's Sudbury mines (including Creighton, 2.4 km deep) since 2015. Sandvik's Newtrax platform monitors Glencore's people and equipment underground. Vale runs a remote operations centre in Sudbury watching all five Ontario mines, with ventilation-on-demand saving millions.

So the data is there. What's missing is the AI interpretation layer that turns raw numbers into a prioritised, plain-English decision, and a practical operator interface built for the person on shift, not just an engineer. OreAcle is that missing layer, built with free open-source AI.

What we use, and how little

Hardware: humble, plug-and-play, in the box

Everything comes from one SunFounder Raspberry Pi Pico W Ultimate Starter Kit. No soldering. Hardware is ~10% of the effort; the idea, the AI, and the interface are the stars.

RP2040

Raspberry Pi Pico W

The edge "nervous system"

  • CPUDual-core Arm Cortex-M0+ @ 133 MHz
  • Memory264 KB SRAM · 2 MB flash
  • Wireless2.4 GHz Wi-Fi (CYW43439)
  • LanguageMicroPython
  • Link to MacUSB serial (primary)

Myth-buster: Amazon lists "Tegra / FreeRTOS", that's an auto-fill error. It's an RP2040 microcontroller. It's too small to run the AI, which is exactly why the brain lives on the MacBook.

Appendix
Anticipated Questions

Q&A

Why not just use GPT-4 or Claude?
Cloud AI requires reliable internet (unavailable underground), sends sensitive operational data off-site, and costs thousands per month at 24/7 multi-zone scale. Llama 3.2 3B on Ollama is free forever, offline, and fast enough for shift-level alerts on commodity hardware.
The Pico W can't run AI — how does it work?
Correct. The Pico W only senses (reads sensors) and acts (drives the fan, LED, LCD). All AI — Isolation Forest, VOD logic, and Ollama LLM — runs on the MacBook. The Pico streams raw JSON over USB; the laptop does all the thinking.
What happens when Ollama isn't running?
The backend has a deterministic template fallback that generates structured, human-readable alerts using the sensor values directly — no LLM required. The ML anomaly detection, VOD logic, and UI all continue to operate normally.
Is the gas detection real?
CO, NO₂, and airflow are software-simulated in this POC. Temperature, humidity, and vibration are real sensors being assembled. Swapping in a ~$10 MQ-7 gas sensor for CO is a single wire change to the Pico W — the backend ingest pipeline is already written to handle it.
How does this scale to a real mine?
The Pico W POC demonstrates the data contract and AI pipeline. A production deployment replaces the Pico W with industrial ARM/ESP32 nodes in IP67 enclosures and swaps USB for LoRa or Wi-Fi mesh. The FastAPI backend, ML models, LLM, and entire UI remain unchanged — only the edge node hardware upgrades.
Appendix
Glossary

Key terms used throughout this presentation.

VOD — Ventilation-on-Demand
Automatically adjusting fan speed to match real-time occupancy and air quality instead of running at a fixed rate. Saves 40–60% ventilation energy in mines.
Isolation Forest
A scikit-learn ML model that detects anomalies by isolating data points that are "different" from learned normal patterns, without needing labelled examples of failures.
Ollama
An open-source tool for running large language models (LLMs) locally on a laptop with no internet, no API key, and no per-token cost. OreAcle uses it to run Llama 3.2 3B.
Llama 3.2 3B
Meta's open-source small language model (3 billion parameters). Fast enough for real-time alerts on a laptop CPU, with quality sufficient for one-sentence mine-safety communications.
Pico W
Raspberry Pi Pico W: a dual-core microcontroller (~$10) with Wi-Fi and MicroPython support. The edge sensor node in OreAcle's hardware POC. Does not run AI — only sense and act.
MPU6050
A 6-axis IMU sensor measuring acceleration and gyroscope data via I²C. OreAcle uses the accelerometer to detect equipment vibration anomalies indicating bearing wear or imminent failure.
DHT11
A basic digital temperature and humidity sensor. Used in the Pico W POC to measure ambient conditions in the simulated mine zone for heat-stress and ventilation effectiveness monitoring.
CO / NO₂
Carbon monoxide and nitrogen dioxide — the two primary gas hazards from diesel engines underground. CO above 25 ppm or NO₂ above 3 ppm triggers OreAcle's WATCH status; hard limits are higher.
WebSocket
A persistent, bidirectional connection between the FastAPI backend and the browser. OreAcle uses it to push every sensor reading to the operator UI in real time without polling.
IROC
Vale's Integrated Remote Operations Centre in Sudbury — oversees all five Ontario underground mines from the surface, proving that remote mine monitoring at scale works today. OreAcle is the open-source AI layer for that paradigm.
